## Что такое поиск вообще

Поиск — это ответ на вопрос: **«Есть ли этот элемент в коллекции, и если да — где?»**

У любого алгоритма поиска есть три характеристики, которые нужно знать наизусть:
1. **Временная сложность** — сколько операций нужно в худшем случае
2. **Пространственная сложность** — сколько дополнительной памяти нужно
3. **Предусловие** — что должно выполняться, чтобы алгоритм работал

## 1. Линейный поиск (Linear Search)

### Интуиция

Представь, что ты ищешь ключ в ящике стола, где вещи свалены как попало. Единственный способ — перебирать по одной, пока не найдёшь. Это и есть линейный поиск: он **ничего не предполагает** о порядке данных.

### Алгоритм пошагово

Дан массив `[34, 7, 23, 32, 5, 62]`, ищем `32`.

```
Шаг 1: смотрим на индекс 0 → 34. Не то.
Шаг 2: смотрим на индекс 1 → 7.  Не то.
Шаг 3: смотрим на индекс 2 → 23. Не то.
Шаг 4: смотрим на индекс 3 → 32. НАШЛИ. Возвращаем 3.
```

### Реализация на JS

```js
function linearSearch(arr, target) {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1; // не найдено
}

linearSearch([34, 7, 23, 32, 5, 62], 32); // → 3
linearSearch([34, 7, 23, 32, 5, 62], 99); // → -1
```

### Big O

| Случай    | Время   | Пространство |
|-----------|---------|--------------|
| Лучший    | O(1)    | O(1)         |
| Средний   | O(n)    | O(1)         |
| Худший    | O(n)    | O(1)         |

**Лучший случай** — элемент стоит первым.
**Худший случай** — элемент последний или его нет вовсе: нужно пройти все n элементов.
**Пространство O(1)** — алгоритм не создаёт новых структур, только хранит переменную `i`.

### Когда использовать

✅ Данные не отсортированы
✅ Массив маленький (до ~100–1000 элементов — разница незаметна)
✅ Ищешь по сложному условию, которое нельзя «отсортировать»

❌ Данные отсортированы — используй бинарный поиск
❌ Поиск повторяется многократно по большому набору — используй хеш-таблицу

## 2. Бинарный поиск (Binary Search)

### Интуиция

Ты ищешь слово в бумажном словаре. Ты не листаешь с первой страницы — ты открываешь середину, смотришь, слово стоит раньше или позже, и выбрасываешь половину словаря. Потом опять делишь пополам. И так до тех пор, пока не останется одна страница.

**Ключевая идея**: каждый шаг уменьшает область поиска вдвое. За 10 шагов можно проверить 1024 элемента. За 20 шагов — больше миллиона.

**Обязательное предусловие**: массив должен быть **отсортирован**.

### Алгоритм пошагово

Дан массив `[2, 5, 8, 12, 16, 23, 38, 56, 72, 91]`, ищем `23`.

```
left = 0, right = 9

Шаг 1:
  mid = Math.floor((0 + 9) / 2) = 4
  arr[4] = 16. 16 < 23 → ищем правее → left = 5

Шаг 2:
  mid = Math.floor((5 + 9) / 2) = 7
  arr[7] = 56. 56 > 23 → ищем левее → right = 6

Шаг 3:
  mid = Math.floor((5 + 6) / 2) = 5
  arr[5] = 23. НАШЛИ. Возвращаем 5.
```

Всего 3 шага вместо 6 (линейный нашёл бы за 6 шагов от начала).

### Реализация на JS — итеративная

```js
function binarySearch(arr, target) {
  let left = 0;
  let right = arr.length - 1;

  while (left <= right) {
    const mid = Math.floor((left + right) / 2);

    if (arr[mid] === target) return mid;
    if (arr[mid] < target) left = mid + 1;
    else right = mid - 1;
  }

  return -1;
}

binarySearch([2, 5, 8, 12, 16, 23, 38, 56, 72, 91], 23); // → 5
binarySearch([2, 5, 8, 12, 16, 23, 38, 56, 72, 91], 99); // → -1
```

### Реализация на JS — рекурсивная

```js
function binarySearchRecursive(arr, target, left = 0, right = arr.length - 1) {
  if (left > right) return -1;

  const mid = Math.floor((left + right) / 2);

  if (arr[mid] === target) return mid;
  if (arr[mid] < target) return binarySearchRecursive(arr, target, mid + 1, right);
  return binarySearchRecursive(arr, target, left, mid - 1);
}
```

Рекурсивная версия элегантнее, но каждый рекурсивный вызов кладёт фрейм на стек вызовов — поэтому пространственная сложность становится O(log n).

### Big O

| Случай    | Время      | Пространство (итерат.) | Пространство (рекурс.) |
|-----------|------------|------------------------|------------------------|
| Лучший    | O(1)       | O(1)                   | O(1)                   |
| Средний   | O(log n)   | O(1)                   | O(log n)               |
| Худший    | O(log n)   | O(1)                   | O(log n)               |

**Почему log n?** Каждый шаг делит область поиска пополам. Если массив из n элементов, максимальное количество шагов — это сколько раз можно делить n на 2, пока не получишь 1. Это и есть log₂(n).

Для n = 1 000 000: log₂(1000000) ≈ 20 шагов. Линейный потребовал бы до миллиона.

### Типичная ошибка: integer overflow

В языках с фиксированными целыми числами вычисление `mid = (left + right) / 2` может переполниться, если left и right большие. Безопасный вариант:

```js
const mid = left + Math.floor((right - left) / 2);
```

В JavaScript числа 64-битные float, поэтому переполнение практически невозможно — но знать этот паттерн нужно для интервью.

### Когда использовать

✅ Массив отсортирован
✅ Данных много (тысячи и больше)
✅ Поиск нужен быстро и не хочется строить хеш-таблицу

❌ Данные не отсортированы (нужна предварительная сортировка — O(n log n), что может быть дороже)
❌ Частые вставки/удаления — при каждом изменении нужно пересортировывать

## 3. BFS — поиск в ширину (Breadth-First Search)

### Интуиция

Представь, что ты стоишь в центре паутины и хочешь найти муху. Сначала ты смотришь на все узлы, непосредственно соединённые с тобой (расстояние 1). Затем — на все, соединённые с ними (расстояние 2). И так далее, слой за слоем. Ты гарантированно найдёшь самый **ближайший** путь.

BFS использует **очередь** (FIFO — первым пришёл, первым ушёл). Ты добавляешь узлы в хвост, берёшь из головы.

### Алгоритм пошагово

Граф:
```
    A
   / \
  B   C
 / \   \
D   E   F
```

BFS от A, ищем F:

```
Очередь: [A]
Посещённые: {A}

Шаг 1: Берём A. Добавляем соседей B, C → Очередь: [B, C]
Шаг 2: Берём B. Добавляем соседей D, E → Очередь: [C, D, E]
Шаг 3: Берём C. Добавляем соседей F   → Очередь: [D, E, F]
Шаг 4: Берём D. Соседей нет.          → Очередь: [E, F]
Шаг 5: Берём E. Соседей нет.          → Очередь: [F]
Шаг 6: Берём F. НАШЛИ.
```

Порядок обхода: A → B → C → D → E → F

### Реализация на JS

```js
function bfs(graph, start, target) {
  const queue = [start];
  const visited = new Set([start]);

  while (queue.length > 0) {
    const node = queue.shift(); // берём из головы

    if (node === target) return true;

    for (const neighbor of graph[node] || []) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor);
        queue.push(neighbor); // добавляем в хвост
      }
    }
  }

  return false;
}

const graph = {
  A: ['B', 'C'],
  B: ['D', 'E'],
  C: ['F'],
  D: [], E: [], F: []
};

bfs(graph, 'A', 'F'); // → true
```

### Big O

| Случай    | Время      | Пространство |
|-----------|------------|--------------|
| Любой     | O(V + E)   | O(V)         |

V — количество вершин (vertices), E — количество рёбер (edges).
Пространство O(V) — в очереди может находиться до всех вершин.

### Когда использовать

✅ Нужен **кратчайший путь** в невзвешенном графе
✅ Нужно найти **ближайшее** что-то (ближайший склад, ближайший друг в соцсети)
✅ Обход дерева **уровень за уровнем** (level-order traversal)

❌ Граф очень глубокий и широкий — очередь может разрастись огромной (DFS лучше)
❌ Нужен произвольный путь, а не кратчайший

## 4. DFS — поиск в глубину (Depth-First Search)

### Интуиция

Представь лабиринт. Ты идёшь вперёд, пока не упрёшься в стену. Тогда возвращаешься на последний перекрёсток и пробуешь другой путь. Ты исследуешь один путь **до конца**, прежде чем переключиться.

DFS использует **стек** — либо явно (итеративная версия), либо стек вызовов (рекурсивная версия).

### Три порядка обхода дерева

```
      1
     / \
    2   3
   / \
  4   5
```

- **Pre-order** (узел → лево → право): 1, 2, 4, 5, 3 — используется для копирования дерева
- **In-order** (лево → узел → право): 4, 2, 5, 1, 3 — для BST даёт **отсортированный** порядок
- **Post-order** (лево → право → узел): 4, 5, 2, 3, 1 — для удаления дерева (сначала удаляем листья)

### Реализация на JS — рекурсивная

```js
function dfs(graph, node, target, visited = new Set()) {
  if (node === target) return true;
  visited.add(node);

  for (const neighbor of graph[node] || []) {
    if (!visited.has(neighbor)) {
      if (dfs(graph, neighbor, target, visited)) return true;
    }
  }

  return false;
}

dfs(graph, 'A', 'F'); // → true
```

### Реализация на JS — итеративная (явный стек)

```js
function dfsIterative(graph, start, target) {
  const stack = [start];
  const visited = new Set();

  while (stack.length > 0) {
    const node = stack.pop(); // берём с вершины стека

    if (node === target) return true;
    if (visited.has(node)) continue;
    visited.add(node);

    for (const neighbor of graph[node] || []) {
      stack.push(neighbor);
    }
  }

  return false;
}
```

### Big O

| Случай    | Время      | Пространство (рекурс.) | Пространство (итерат.) |
|-----------|------------|------------------------|------------------------|
| Любой     | O(V + E)   | O(V) — стек вызовов    | O(V) — явный стек      |

### Когда использовать

✅ Нужно **проверить связность** графа
✅ Поиск пути в лабиринте
✅ Топологическая сортировка (DAG)
✅ Обнаружение циклов в графе
✅ Глубина рекурсии не критична (дерево не бесконечно)

❌ Нужен **кратчайший путь** — BFS надёжнее
❌ Граф очень глубокий → риск переполнения стека вызовов (stack overflow) при рекурсии

## Сравнительная таблица

| Алгоритм       | Структура данных | Предусловие  | Время       | Лучше всего для                         |
|----------------|-----------------|--------------|-------------|----------------------------------------|
| Linear Search  | Массив/список   | Нет          | O(n)        | Несортированные данные, простой поиск  |
| Binary Search  | Массив          | Отсортирован | O(log n)    | Большие отсортированные наборы         |
| BFS            | Граф/дерево     | Нет          | O(V + E)    | Кратчайший путь, поиск ближайшего      |
| DFS            | Граф/дерево     | Нет          | O(V + E)    | Обход, цикл, связность, топосортировка |
